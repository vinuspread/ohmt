# PRD: 템플릿허브 내장형 템플릿 등록/미리보기/신청 서비스

## 1) 문서 목적
- 현재 논의된 기획안을 제품 요구사항으로 정리한다.
- 대상 기능을 **관리자(Admin)** / **사용자(User)** / **내보내기(Export)** 분기로 명확히 나눈다.
- 본 문서는 **설계 기준**이며, 구현 상세(코드/DB 마이그레이션)는 별도 문서에서 다룬다.

---

## 2) 배경 및 문제
- 템플릿허브에는 여러 템플릿이 존재하며, 사용자는 템플릿 화면을 확인한 뒤 신청하고자 한다.
- 현재 템플릿 목록/노출 기준이 일부 하드코딩되어 있고, 신청 흐름은 표준화되어 있지 않다.
- 운영자가 템플릿을 쉽게 추가/관리할 수 있는 관리자 기능이 필요하다.

---

## 3) 핵심 원칙 (중요)
1. **내장형 only**: 모든 템플릿은 템플릿허브 내부에서 실행된다. (외부 실행/외부 연동 없음)
2. **관리자 중심 등록**: 템플릿 등록/수정/삭제/노출 제어는 관리자에서 수행한다.
3. **사용자 흐름 단순화**: 사용자 흐름은 `탐색 → 미리보기 → 신청`으로 통일한다.
4. **분류와 실행 분리**: `JS 유형(Next.js / React.js)`은 관리 분류값이며, 실행은 모두 템플릿허브 내부에서 수행한다.

### 3-1) 구현 가드레일 (중요)
1. 각 기능은 **기능 단위로 독립 파일**로 구성한다.
2. **UI 요소와 개발(비즈니스/데이터) 요소를 분리**한다.
3. 관리자 UI는 `admin-designsystem.json`을 기준으로 구성한다.
4. `PRD.md`에 명시되지 않은 기능은 임의로 구현하지 않는다.

적용 메모:
- `admin-designsystem.json` 파일은 루트 경로(`/admin-designsystem.json`)에 존재한다.

### 3-2) 예외 규칙 (긴급 변경 허용 범위)
`PRD.md` 외 기능은 기본적으로 금지한다. 단, 아래 조건을 모두 만족할 때만 예외를 허용한다.

허용 조건:
1. 서비스 장애 또는 기능 중단(신청 불가, 관리자 핵심 작업 불가)
2. 보안 취약점 또는 데이터 손상 위험
3. 법적/운영상 즉시 조치가 필요한 문제

예외 작업 원칙:
1. **기존 기능 복구 목적의 최소 수정만** 허용
2. 신규 기능 추가, 화면 확장, 정책 변경은 금지
3. 예외 작업은 임시 조치로 간주하며 사후 정식 반영 여부를 결정

기록 의무:
- 예외 ID
- 발생 시각/영향 범위
- 원인 및 수정 내용
- 롤백 방법
- 후속 조치 일정

승인/처리:
- 관리자 승인 후 적용
- 적용 후 즉시 운영 로그에 기록

### 3-3) 백업 계획 (운영 필수)
본 서비스는 변경/배포 전후로 데이터와 산출물을 백업해야 한다.

백업 대상:
1. DB 데이터
   - 템플릿(`Template`), 템플릿 유형(`TemplateType`), 신청(`TemplateRequest`), 내보내기(`TemplateExportJob`)
2. 파일 데이터
   - 썸네일/미리보기 이미지
   - Export 결과물(ZIP)
3. 설정/운영 문서
   - `PRD.md`, `admin-designsystem.json`, 환경 설정 템플릿(`.env.example`)

백업 실행 방식:
1. 관리자 수동 백업
   - 관리자 화면에서 **버튼 1회 클릭으로 즉시 백업** 실행 가능해야 한다.
2. 정기 백업
   - 6시간 단위 고정 스케줄로 실행한다.
   - 실행 시각: **오전 10시 / 오후 4시 / 저녁 10시 / 새벽 4시**
   - 위 스케줄을 정기 백업 기준으로 사용한다.

보관/복구 기준:
1. 보관 정책
   - 백업 파일명 규칙: `backup+날짜+시간`
   - 예시: `backup_20260408_1000`, `backup_20260408_2200`
   - 백업 파일은 **항상 4개만 유지**한다.
   - 신규 백업 파일 생성 시 가장 오래된 백업 파일 1개를 자동 삭제한다.
2. 복구 목표
   - RPO: 최대 1시간 이내 데이터 손실
   - RTO: 4시간 이내 핵심 기능 복구

복구 절차(요약):
1. 장애 등급 판단 및 복구 책임자 지정
2. 최근 정상 백업 선택
3. DB 복구 → 파일 복구 → 서비스 검증 순으로 진행
4. 신청/템플릿 조회 및 관리자 핵심 동작 검증
5. 복구 완료 로그 및 원인 분석 리포트 작성

복구 실행 방식 (운영 규칙):
1. 관리자에서 백업 파일을 선택해 **복구 버튼 1회 클릭**으로 복구를 시작할 수 있어야 한다.
2. 복구 시작 직전에 현재 상태를 임시 백업(스냅샷)으로 1회 저장해야 한다. (롤백 대비)
3. 복구 대상은 백업 파일에 포함된 DB/파일 세트를 동일 시점 기준으로 적용한다.
4. 복구 완료 후 자동 검증 항목을 수행한다.
   - 템플릿 리스트 조회 정상
   - 신청 목록 조회 정상
   - 관리자 주요 화면 진입 정상
5. 복구 실패 시 즉시 직전 임시 백업으로 롤백할 수 있어야 한다.

### 3-4) 템플릿 표준 디렉토리 구조 (확정)
관리자 업로드/등록/내보내기 자동화를 위해 템플릿 디렉토리 구조를 아래로 확정한다.

기본 원칙:
1. 템플릿마다 구조가 제각각이면 운영 자동화가 불가능하므로, **신규 템플릿은 표준 구조를 강제**한다.
2. 기존 레거시 템플릿은 단계적 마이그레이션 대상으로 관리한다.
3. `templateType`(홍보형/쇼핑몰형/기타)는 분류값일 뿐이며, **ZIP 디렉토리 구조에는 영향을 주지 않는다.**
4. 즉, 타입이 달라도 업로드 ZIP 구조는 100% 동일해야 한다.

#### (A) 템플릿허브 내부 표준 구조 (신규 필수)
`src/app/{templateSlug}/`

```text
src/app/{templateSlug}/
  page.tsx                     # 필수: 템플릿 메인 페이지
  theme.json                   # 필수: 템플릿 테마 설정
  components/                  # 필수: 템플릿 전용 UI 컴포넌트
    layout/
    sections/
    ui/
  lib/                         # 선택: 데이터/유틸
    data.ts
  context/                     # 선택: 상태 관리
  [subroute]/page.tsx          # 선택: 서브 페이지들
```

금지 규칙(신규 기준):
1. 템플릿 전용 컴포넌트를 `src/components/{templateSlug}`에 신규 생성 금지
2. `src/app/{templateSlug}/src/components` 같은 중첩 `src/` 패턴 금지
3. 동일 목적의 설정 파일 중복 금지 (`theme.json` 1개 기준)

#### (B) 관리자 업로드 ZIP 표준 구조 (신규 필수)
업로드 파일 확장자는 `.zip`이며, 내부 구조는 아래를 따른다.
아래 구조는 모든 템플릿 타입에 공통으로 적용한다.

```text
{templateSlug}.zip
  template/
    manifest.json              # 필수: 메타/경로 계약
    app/
      page.tsx                 # 필수
      theme.json               # 필수
      components/              # 필수
      lib/                     # 선택
      context/                 # 선택
      subroutes/               # 선택
    assets/
      thumbnails/              # 필수: 기본 3장 이상
```

`manifest.json` 필수 키:
- `name`, `slug`, `jsType`, `templateType`
- `entry`: `app/page.tsx`
- `theme`: `app/theme.json`
- `thumbnailPaths`: 최소 3개

#### (C) 관리자 검증 규칙 (업로드 단계)
1. ZIP 구조/필수 파일 누락 시 업로드 거부
2. 썸네일 3장 미만 시 업로드 거부
3. `manifest.json`의 경로와 실제 파일 경로 불일치 시 업로드 거부
4. 검증 통과 시에만 등록/내보내기 가능
5. 타입별 예외 구조는 허용하지 않으며, 타입을 이유로 한 경로 변형 시 업로드 거부

#### (D) 레거시 전환 정책
1. 현재 운영 중 템플릿은 레거시 구조를 임시 허용한다.
2. 신규 등록 템플릿부터 표준 구조 강제 적용한다.
3. 레거시 템플릿은 우선순위에 따라 표준 구조로 순차 전환한다.

### 3-5) 관리자 인증 정책 (Google + 환경변수 allowlist)
관리자 인증은 Google 로그인 기반으로 수행하며, 관리자 허용 계정은 환경변수 allowlist로 관리한다.

규칙:
1. 로컬 개발 환경에서는 `.env.local`에 관리자 이메일을 등록한다.
2. 인증 시 로그인한 Google 이메일이 allowlist에 포함된 경우에만 관리자 접근을 허용한다.
3. allowlist에 없는 이메일은 관리자 접근을 거부한다.

환경변수:
- `ADMIN_ALLOWLIST_EMAILS` (콤마 구분)
- 예시: `ADMIN_ALLOWLIST_EMAILS=admin1@gmail.com,admin2@gmail.com`

보안 원칙:
1. 실제 관리자 이메일 값은 문서에 기록하지 않는다.
2. 운영 환경에서는 `.env.local` 대신 배포 환경변수(Vercel Environment Variables)로 동일 키를 설정한다.
3. allowlist 검증은 서버에서 수행한다.

## 6) 기능 요구사항

## A. 관리자 기능

### A-1. 템플릿 관리 (추가/수정/삭제)
관리자는 템플릿 정보를 CRUD 할 수 있어야 한다.

### A-1-1. 템플릿 설명필드 표준안 (우리 서비스 기준)
본 서비스는 **웹 중심(현재 앱 제외)**이며, 템플릿은 주로 **메인 + 서브 4~5개 페이지**로 구성된
정보제공형/쇼핑형 사이트를 대상으로 한다. 따라서 설명필드는 아래를 기본 표준으로 사용한다.

#### (1) 목록 카드용 필드 (빠른 탐색)
- 템플릿명
- 썸네일(기본 3장 중 대표 1장 노출)
- 한줄 요약 (최대 80자)
- 템플릿 유형: 기본 제공 5종(`홍보형` | `쇼핑몰형` | `비즈니스형` | `개인형` | `관리형`) + 관리자 추가 유형
- JS 유형: `Next.js` | `React.js`
- 노출 상태: `공개` | `비공개`
- 신청 가능 여부: `ON` | `OFF`

#### (2) 상세 설명용 필드 (의사결정)
- 상세 요약 설명 (최대 300자)
- 추천 사용 업종/시나리오 (태그)
- 페이지 구성 정보
  - 메인 페이지 포함 여부 (항상 true)
  - 서브 페이지 수 (`4~5` 권장, 실제 값 저장)
  - 대표 페이지 목록 (예: About, Services/Products, FAQ, Contact)
- 핵심 기능 요약
  - 홍보형 예: 브랜드 소개, 핵심 메시지, CTA, 문의 유도
  - 쇼핑몰형 예: 상품목록, 상품상세, 장바구니, 결제 유도 CTA
  - 비즈니스형 예: 서비스 소개, 사례/레퍼런스, 문의/상담 폼
  - 개인형 예: 프로필, 포트폴리오, 이력/연락처
  - 관리형 예: 대시보드, 데이터 목록, 상태/통계, 액션 버튼
- 미리보기 자산
  - 기본 썸네일 3장(필수)
  - 추가 썸네일 업로드(선택, 다건 가능)
  - 목록 카드에는 기본 3장 중 대표 1장을 사용

#### (3) 운영/정렬 필드
- 정렬 순서 (`최신순` | `인기순`)
- 버전/개정 메모
- 최근 업데이트 일시

#### (4) 필드 설계 원칙
- 설명필드는 **사용자 선택에 필요한 정보** 중심으로 최소화한다.
- 개발/로컬 실행 전용 값(예: 로컬 포트, 로컬 URL)은 사용자 노출용 설명필드에서 제외한다.
- 정보제공형/쇼핑형 공통 비교가 가능하도록 동일 키 구조를 유지한다.
- 템플릿 유형은 고정 enum이 아닌 **관리 가능한 마스터 데이터**로 운영한다.

### A-1-2. 템플릿 유형 관리 (추가/수정/비활성화)
관리자는 템플릿 유형(Type)을 별도로 관리할 수 있어야 한다.

기본 제공 유형(초기값):
- 홍보형
- 쇼핑몰형
- 비즈니스형
- 개인형
- 관리형

관리 기능:
- 유형 추가
- 유형명 수정
- 유형 비활성화(삭제 대체)
- 유형 정렬 순서 관리

제약:
- 유형명 중복 불가
- 이미 템플릿에 연결된 유형은 하드 삭제 금지 (비활성화만 허용)
- 비활성화된 유형은 신규 템플릿 등록 시 선택 불가

필수 입력 항목:
- 템플릿명
- slug (URL 식별자, 시스템 자동 생성)
- 요약 설명
- 기본 썸네일 3장
- JS 유형: `Next.js` | `React.js`
- 노출 상태: `공개` | `비공개`
- 신청 가능 여부: `ON` | `OFF`

권장 입력 항목:
- 태그
  - 입력 방식: `,`(콤마)로 구분 입력
  - 활용 방식: 목록/검색 필터에 활용
  - 관리 기능: 추가/수정/삭제 필수
- 버전/개정메모
  - 입력 방식: 1줄 1항목
  - 노출 방식: 버전/개정 이력 영역에 그대로 노출
  - 관리 기능: 추가/수정/삭제 필수
- 정렬순서
  - 기준값: `최신순` | `인기순`

제약:
- slug는 `템플릿명 정규화값 + 난수 5자리` 규칙으로 자동 생성한다. (관리자 수동 입력 없음)
- 예시: `fashion-mall-a3k9q`, `studio-lite-x7m2p`
- slug는 **영문 소문자, 숫자, 하이픈(-)** 만 허용한다. (대문자 사용 불가)
- slug 중복 불가 (생성 시 유니크 검증 및 충돌 시 재생성)
- 삭제 시 소프트 삭제 우선 (히스토리/신청 데이터 보존)
- 태그/버전·개정메모는 운영 중 수정/삭제가 가능해야 한다.

### A-2. 템플릿 노출 제어
- 공개 상태인 템플릿만 사용자 허브 목록에 노출한다.
- 신청 가능 OFF인 템플릿은 미리보기는 가능하되 신청 버튼은 비활성 처리한다.

### A-3. 신청 관리
- 신청 목록 조회/검색/필터링 가능해야 한다.
- 상태값으로 관리한다: `접수` → `검토중` → `완료` (필요 시 `보류`)
- 신청 상세에서 메모 및 처리 이력 기록 가능해야 한다.

### A-4. 기본 운영 지표 (초기)
- 기간별 신청 수
- 템플릿별 신청 수
- 처리 상태별 건수

## B. 사용자 기능

### B-1. 템플릿 탐색
- 사용자는 템플릿허브에서 공개된 템플릿 목록을 본다.
- 카드 정보: 썸네일, 템플릿명, 요약 설명, 태그(선택)
- 현재 구현 확인: 사용자 템플릿 리스트 화면은 `src/app/page.tsx`에서 동작 중이다.

### B-2. 템플릿 미리보기
- 템플릿 상세/실행 화면에서 UI를 확인한다.
- 모든 미리보기는 템플릿허브 내부 라우트에서 동작한다.

### B-3. 신청하기
- 사용자는 템플릿별 `신청하기` 버튼으로 신청한다.
- 신청 폼 필수 항목(초안):
  - 이름
  - 연락처(이메일 또는 전화)
  - 회사/브랜드명 (선택)
  - 요청 내용
  - 관심 템플릿 (자동 바인딩)
- 제출 후 성공/실패 피드백을 즉시 제공한다.

### B-4. 신청 제한 정책
- 신청 가능 OFF 템플릿은 신청 폼 접근 차단
- 비공개 템플릿은 사용자 경로에서 접근 불가

## C. 내보내기(Export) 기능

### C-1. 기능 목적
- 사용자가 신청을 완료하면, 운영자는 해당 신청 건 기준으로 템플릿을 외부 프로젝트로 내보낼 수 있어야 한다.
- 내보내기 결과물은 **해당 템플릿에 필요한 요소/기능만 포함한 독립 프로젝트**여야 한다.
- 기본 기술 기준은 **Vercel + Supabase**로 통일한다.

### C-2. 내보내기 트리거 및 권한
- 트리거: `TemplateRequest`가 `RECEIVED` 이상일 때 관리자에서 내보내기 생성 가능
- 권한: 관리자만 내보내기 실행 가능
- 정책: 1개 신청 건당 복수 Export 버전 생성 가능(이력 관리)

### C-3. 내보내기 대상 범위 (필수)
내보내기 결과물에는 아래만 포함한다.
1. 선택된 템플릿의 라우트/페이지
2. 선택된 템플릿이 직접 참조하는 컴포넌트/유틸/스타일
3. 선택된 템플릿 전용 에셋
4. 해당 템플릿 실행에 필요한 최소 공통 의존성

반드시 제외한다.
1. 다른 템플릿 폴더 및 코드
2. 허브 전용 관리자/운영 코드
3. 사용되지 않는 공통 컴포넌트/자산

### C-4. 결과물 형태
- 1차 결과물: 다운로드 가능한 ZIP 아카이브
- ZIP 내부 프로젝트는 즉시 배포 가능한 형태여야 한다.

기본 포함 파일(최소 규격):
- `package.json`
- `next.config.*`
- `tsconfig.json`
- `vercel.json`
- `.env.example` (Supabase 키 이름 포함, 실제 값 제외)
- 템플릿 실행 코드(`src/app/...` 또는 동등 구조)
- `README.md` (실행/배포 가이드)

### C-5. Vercel + Supabase 기준 규격
- 프레임워크: Next.js 기반으로 배포 가능한 구조
- 배포 타겟: Vercel 기본 설정 포함
- 데이터/인증 백엔드: Supabase 연결 전제

환경변수 규격(초안):
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY` (필요 기능이 있을 때만)

보안 규칙:
- 실제 시크릿은 절대 내보내지 않음
- `.env.example`에는 키 이름만 제공
- 민감정보/운영 토큰/내부 로그는 결과물에서 제거

### C-6. 신청-내보내기 사용자/운영 흐름
1. 사용자가 템플릿 미리보기 후 신청 제출
2. 관리자가 신청 검토 후 내보내기 생성 실행
3. 시스템이 템플릿 의존성 그래프를 기준으로 최소 파일 세트 추출
4. Vercel + Supabase 기본 구성 주입 후 ZIP 생성
5. 관리자에서 다운로드 링크 확인 및 전달

### C-7. 실패/예외 처리
- 필수 파일 누락 시 Export 실패 처리 및 사유 표시
- Supabase 관련 환경변수 요구사항을 README에 명시
- 내보내기 생성 실패 시 재시도 가능해야 함

- 운영자가 코드 수정 없이 템플릿 등록/노출 제어 가능

정량 예시:

- 신청 제출 성공률 99%+
- 신청 상태 처리 리드타임 측정 가능
- Export 성공률 95%+
- Export 결과물 생성 시간(중간 크기 템플릿) 5분 이내

## 10) 수용 기준 (Acceptance Criteria)
1. 관리자가 새 템플릿을 등록하면, 공개 상태에서 사용자 허브에 노출된다.
2. 관리자가 템플릿을 비공개 처리하면 사용자 허브/직접 접근에서 차단된다.
3. 사용자가 템플릿 미리보기 후 신청 폼 제출 시 신청 데이터가 저장된다.
4. 관리자가 신청 상태를 변경하면 목록/상세에 즉시 반영된다.
5. 신청 가능 OFF 템플릿은 사용자 신청이 불가능하다.
6. 관리자가 신청 건을 선택해 Export를 실행하면 ZIP 결과물이 생성된다.
7. Export ZIP에는 선택 템플릿 실행에 필요한 파일만 포함되며, 다른 템플릿 코드는 제외된다.
8. Export 결과물은 Vercel 배포 및 Supabase 환경변수 연결 가이드를 포함한다.
9. 관리자가 템플릿 유형을 추가/수정/비활성화할 수 있고, 비활성 유형은 신규 등록에서 선택되지 않는다.
10. 템플릿 등록 시 기본 썸네일 3장이 필수이며, 추가 썸네일을 다건으로 등록할 수 있다.
11. 태그와 버전/개정메모는 관리자에서 생성 후 수정/삭제할 수 있고, 변경사항은 즉시 반영된다.
12. 관리자는 버튼 1회 클릭으로 수동 백업을 실행할 수 있어야 한다.
13. 정기 백업은 6시간 단위(10:00, 16:00, 22:00, 04:00)로 자동 실행되어야 한다.
14. 백업 파일은 `backup+날짜+시간` 규칙으로 생성되고, 항상 최신 4개만 유지되어야 한다.
15. 관리자는 백업 파일 선택 후 복구를 실행할 수 있고, 복구 전 임시 백업 생성 및 복구 후 자동 검증이 수행되어야 한다.
16. 신규 템플릿 업로드는 표준 디렉토리 구조(3-4)를 통과해야 하며, 비표준 구조는 등록할 수 없다.
